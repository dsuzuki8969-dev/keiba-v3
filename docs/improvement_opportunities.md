# 精度向上のための情報・改善機会

不足している情報と、入れれば劇的に精度が高くなる情報を、根拠とともに整理した。

---

## 凡例

- **根拠**: コード上の証拠・現状の扱い
- **影響**: どの項目に効くか
- **取得難易度**: 現実的な実装負荷

---

## 1. 劇的に効く（最優先）

### 1-1. course_db への**前半3F（first_3f_sec）**の追加

| 項目 | 内容 |
|------|------|
| **現状** | course_db_collector はレース結果ページから 1〜3着の PastRun を生成するが、**ペース列をパースしていない**。first_3f_sec は常に None。馬の過去走（horse/result）では netkeiba が ペース列 "35.7-36.0" を解析して first_3f を取得している。 |
| **根拠** | `pace_course.py` L69-78: 逃げ馬の過去前半3F が 35.5秒未満なら base += 2（ハイペース寄り）。first_3f_list が空だとこの補正が一切動かない。<br>`pace_analysis.py` L42-45: estimate_pace_times_from_runs で first_3f_list が空なら front=36.0 のデフォルト固定。 |
| **影響** | ペース予測（F-1）、前半/後半3F推定、展開コメント、有利脚質判定 |
| **取得（2通り）** | **A. レース結果テーブル**: netkeiba のレース結果ページの馬別テーブルには**ペース列が存在しない**（列: 着順,枠,馬番,性齢,斤量,騎手,タイム,着差,人気,単勝オッズ,後3F,コーナー通過順,厩舎,馬体重）。馬ごとの前半3F はこのテーブルからは取得不可。<br>**B. ラップタイムセクション（推奨）**: 同一ページの「ラップタイム」に「ペース:H/M/S」と区間ラップ（0-200, 200-400, 400-600...）がある。最初の3区間（0-600m≒前半3F）を合算して first_3f_sec を算出し、**1〜3着の PastRun に共通値として付与**する。馬ごとの値ではないが、レースペースの実測値としてペース予測・前半/後半推定の精度向上に大きく寄与する。 |

### 1-3. **レース単位のラップ／ハロン**

| 項目 | 内容 |
|------|------|
| **現状** | 取得していない。前半3F・上がり3F は「馬ごと」の値。レース全体としての 1000m 通過タイム、各ハロンのラップなどは未使用。 |

## 5. 競馬理論からの補足

- **前半3F**: 「逃げが切るか」「ハイペースになるか」の判断に直接使う。これが欠けていると、ペース予測は脚質の数のみに依存し、質（速さ）を反映できない。
- **コース×脚質**: 差し馬が有利なコース（長直線）と、先行馬が有利なコース（短直線・急坂）は実データで大きく異なる。ルールだけではコースごとの差を吸いきれない。
- **枠番**: 競馬場によって「内枠有利」「外枠有利」が異なる。ダートは外、芝は内が多いが、コース距離や形状でも変わる。データで検証するのが望ましい。
